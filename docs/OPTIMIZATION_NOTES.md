# OhMyUsage 优化方向备忘录

更新时间：2026-05-05

本文档用于记录大版本重构前已经确认过的主要问题、优化方向和产品取向。它不是具体实施计划，而是后续拆方案、排优先级和评审设计时的基础材料。

## 总体目标

将项目从“多个 Provider 的余额读取工具”升级为一个低能耗、可扩展、可信度明确的 AI 订阅与额度控制台。

核心原则：

- 常驻菜单栏应用必须省电、安静、稳定。
- 首页只回答高频问题：当前能不能用、哪个账号在用、有没有异常。
- 复杂数据进入设置页或详情页，不在菜单栏首屏平铺。
- 所有额度、倒计时、余额和用量数据都要标注来源、时间和可信度。
- Provider、账号、历史用量、刷新调度、UI 展示模型需要逐步解耦。

## 1. 架构解耦

当前问题：

- `AppViewModel` 负责过多：配置、刷新、账号切换、Provider 状态、历史用量、权限、更新、UI 状态都集中在一起。
- `SettingsView` 和 `MenuContentView` 过大，业务规则、状态推导和 SwiftUI 展示混在同一层。
- Provider 元信息散落在 factory、菜单栏、设置页、状态栏展示等多个位置。

优化方向：

- 拆出刷新调度、账号状态、历史用量、配置管理、Provider 管理等独立服务。
- UI 层改为消费展示模型，例如 `ProviderCardDisplayModel`、`QuotaMetricDisplayModel`、`UsageHistoryDisplayModel`。
- Provider 能力统一描述：是否支持余额、额度窗口、账号切换、历史用量、浏览器桥接、第三方站点余额等。
- Provider 视觉元信息集中管理，避免图标、名称、颜色、显示规则重复定义。

## 2. 菜单栏界面展示

当前问题：

- 菜单栏像“额度卡片列表”，所有卡片视觉权重接近。
- 当前账号不够突出，其他账号和当前账号信息密度相同。
- 正常状态的“充足”、百分比、进度条重复出现，增加扫描负担。

优化方向：

- 首页改为“状态驾驶舱”：当前账号主卡 + 其他账号紧凑切换器 + 异常摘要。
- 当前账号突出展示，非当前账号降级为紧凑列表。
- 正常状态降噪；异常、过期、限流、认证失败、数据估算时才强化颜色。
- 第三方中转站、新 API、官方订阅采用统一信息语法：服务、账户、余额/额度、重置/到期、可信度。
- 倒计时和额度窗口必须显示数据可信度，例如“官方确认”“本地估算”“待刷新”。

可探索的信息架构：

- 总览驾驶舱：默认首页，快速判断是否正常。
- 重置时间线：展示各服务窗口何时重置。
- 命令面板：用于账号切换、刷新、校准、桥接检查。
- 额度账本：用于第三方中转站、NewAPI、余额、订阅费用和有效期统一监控。

## 3. 钥匙串与浏览器信息获取

当前问题：

- 覆盖安装后可能多次触发钥匙串密码输入。
- 直接读取浏览器 Cookie 或本地凭据容易受到签名、权限和系统安全策略影响。

优化方向：

- 优先保证签名和钥匙串访问组稳定，减少覆盖安装后的重复授权。
- 对本机已授权凭据做短期安全缓存，避免同一轮刷新重复访问钥匙串。
- 长期方案引入浏览器扩展桥接：
  - 浏览器扩展负责在浏览器侧读取授权后的会话信息。
  - App 通过本地安全通道接收扩展传来的最小必要数据。
  - 通信需要来源验证、可撤销授权、最小权限和本地加密。
  - App 不直接长期保存浏览器 Cookie 原文，只保存必要 token 摘要或短期会话材料。

## 4. 第三方中转站与 NewAPI 监控

当前问题：

- 第三方站点差异大，有 NewAPI 框架、二开框架和完全自定义接口。
- 余额、用量、有效期、账号名、分组、渠道等字段不统一。
- 失败原因需要区分认证失败、限流、结构变化、余额接口不可用等。

优化方向：

- 抽象为：`Source Instance = Site Adapter + Auth Policy + Channel Set + Override Rules`。
- 一个站点实例可以拥有多个数据通道，例如余额、token 用量、账单用量、账户信息、有效期。
- Adapter 负责默认路径、字段提取、认证方式和后处理。
- Override Rules 允许用户覆盖 header、路径、JSON 表达式、单位、账号标签。
- 每个站点保存健康状态、上次成功时间、缓存回退状态、适配器模板版本。
- UI 统一展示为“额度账本”，避免每个第三方站点都写一套特殊 UI。

## 5. 账号切换

当前问题：

- 当前账号切换更像单个按钮动作，缺少事务边界。
- 切换过程中容易出现 UI 状态、桌面端实际状态、额度快照不同步。
- 并发切换、切换失败和回滚策略需要更明确。

优化方向：

- 账号切换升级为事务流程：
  - `prepare`：检查目标账号、凭据、目标应用状态。
  - `commit`：写入本地账号配置。
  - `restart/apply`：重启或通知目标应用应用变更。
  - `verify`：确认当前账号确实切换成功。
  - `finalize`：刷新快照、更新 UI、记录结果。
- 增加 provider 级切换锁，避免并发切换。
- 失败时提供明确原因和可回滚状态。
- UI 展示“准备中、切换中、验证中、失败、已完成”，而不是只显示一个按钮。

## 6. 窗口限额倒计时

当前问题：

- 本机倒计时经常不准。
- 官方服务端重置额度后，本机状态不能实时感知。
- 切换账号后窗口计时可能重新刷新，导致用户误判。

优化方向：

- 每个额度窗口增加元数据：
  - `resetSource`：官方、网页观测、本地估算、用户校准。
  - `observedAt`：该窗口何时被观测。
  - `serverClockSkew`：服务端时间偏差。
  - `confidence`：可信度。
  - `windowIdentity`：窗口身份，用于判断是不是同一个重置周期。
- UI 显示可信度标签：“官方确认”“本地估算”“待刷新”。
- 账号切换、官方刷新、服务端时间变化、窗口身份变化都要触发重算。
- 本机估算只能作为 fallback，不能伪装成官方确认数据。

## 7. 历史用量展示与性能

当前问题：

- 历史用量状态目前放在设置页临时状态里，关闭或切换页面后容易重新加载。
- Codex、Claude、Kimi 等本地日志扫描耗时，重复扫描会拖慢设置页。
- 当前已能算 `byModel`，但缓存时被裁剪掉；实际已确认 v1 先展示服务维度。

已确认取向：

- 设置页为主要展示位置，菜单栏不承载历史用量详情。
- 默认展示服务维度，例如 Codex / Claude / Kimi 各用了多少 Token。
- 采用“先快后全”：先显示缓存，后台补充刷新。
- 使用本机持久化聚合缓存，不保存原始聊天内容。

优化方向：

- 新增 app 级 `LocalUsageHistoryRepository`，不要把历史用量缓存放在 `SettingsView`。
- 增加磁盘缓存，例如 `Application Support/OhMyUsage/local_usage_history_cache.json`。
- 缓存 key 包含 provider、scope、account identity，避免全量和按账号互相污染。
- 使用 source fingerprint 判断日志是否变化：来源路径、文件数量、总大小、最新修改时间。
- fingerprint 未变化时直接复用缓存，避免 JSONL 全量重扫。
- 刷新失败时保留旧数据，并标注“刷新失败但显示旧缓存”。

## 8. 低能耗与小内存

当前问题：

- 多 provider 常驻轮询会带来持续网络和 CPU 开销。
- 菜单栏和设置页存在 1 秒级 UI timer。
- 壁纸跟随模式有 0.5 秒探测 timer。
- 本地 session 监控和历史用量扫描会反复枚举文件、解析 JSONL。

已确认取向：

- 优先降低能耗，其次降低内存和提高打开速度。
- 默认采用智能刷新，不追求秒级实时。

优化方向：

- 引入 `RefreshScheduler`，替代每个 provider 一个常驻轮询任务。
- 刷新分层：
  - 菜单栏可见 provider：较积极刷新。
  - 后台 provider：拉长间隔。
  - 历史用量：按需刷新。
  - 失败 provider：指数退避。
- 引入 `ResourceMode`：
  - 标准模式。
  - 低能耗模式。
  - 实时模式。
- UI timer 只在视图可见、倒计时确实需要时启动。
- 壁纸探测改为系统通知 + 打开菜单时校准，避免长期 0.5 秒轮询。
- 本地文件扫描改为文件变化触发 + debounce。
- 所有缓存都要有 TTL、容量上限和裁剪策略。

当前收口：

- 已补最小展示层 smoke suite：
  - [PresentationSmokeTests.swift](../Tests/CraftMeterTests/PresentationSmokeTests.swift)
- 已补设置页生命周期相关单测：
  - [SettingsWindowAppearanceControllerTests.swift](../Tests/CraftMeterTests/SettingsWindowAppearanceControllerTests.swift)
  - [VisibleClockControllerTests.swift](../Tests/CraftMeterTests/VisibleClockControllerTests.swift)
- smoke 覆盖目标是“代表性展示组合不崩、不重复调度、关键文案和输入模型可生成”，不是硬 wall-clock 性能阈值。

## 建议优先级

P0：先解决能直接影响体验的问题：

- 历史用量缓存迁出 `SettingsView`。
- 设置页打开后复用缓存，不重复扫描。
- 菜单栏展示降噪，当前账号突出。
- 倒计时增加可信度标签。

P1：大版本核心架构：

- `RefreshScheduler`。
- `LocalUsageHistoryRepository`。
- Provider 能力模型和展示模型。
- 账号切换事务化。

P2：长期扩展能力：

- 浏览器扩展桥接。
- 第三方站点 adapter 系统。
- 额度账本和重置时间线。
- 后台刷新设置项和运行概览面板。

## 验收标准

- 关闭再打开设置页，历史用量应立即显示缓存，不重新从零加载。
- 菜单栏打开时能在 2 秒内判断当前账号、额度状态和异常状态。
- 后台运行时没有高频无意义 timer。
- 多个 enabled provider 不再全部以固定短间隔常驻轮询。
- 失败、限流、认证失效、缓存回退在 UI 上有明确区分。
- 任何估算数据都不能以官方确认数据的方式展示。
